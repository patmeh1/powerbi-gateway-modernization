Reference Architecture & Use Case

Power BI Gateway Modernization for Microsoft Fabric

Modernize and standardize Power BI Gateway architecture to securely and sustainably connect on-premise data sources to Microsoft Fabric, while reducing technical debt and improving scalability.

Core use case: Move away from legacy, unsupported on-prem infrastructure (Windows Server 2012 + older SQL), re-architect Power BI Gateway connectivity to support Fabric / cloud analytics, and reduce operational complexity from many point-to-point gateway connections.

The problem being solved

What surfaced in the customer conversation — the current Power BI Gateway estate is fragmented, running on unsupported infrastructure, and is becoming a bottleneck for Fabric adoption.

Current state pain points

  • Legacy Gateways running on Windows Server 2012 (out of support)
  • Shared Installed on multi-purpose machines, mixing prod & non-prod workloads
  • Sprawl Many one-by-one connections through gateways with no clear pattern
  • Governance No standard architecture, ownership, or environment alignment
  • Risk Security, stability, and compliance exposure on aging SQL infrastructure

Constraints that must be respected

  • Hybrid On-prem connectivity must remain — SQL Server, AD, Linux/file shares
  • Compliance Healthcare data residency & HIPAA controls in scope
  • Continuity Active reporting workloads cannot be disrupted during migration
  • Footprint Hybrid architecture is the target for the foreseeable future
  • Cost Customer is evaluating internal effort vs. Microsoft-assisted delivery

The use case, broken down

Six business + technical workstreams that fall out of the core modernization use case.

1

Hybrid analytics enablement

Allow Power BI / Fabric in Azure to access on-prem SQL Server databases, file shares (Linux / network drives), and AD-based data sources via the On-Premises Data Gateway.

2

Legacy infrastructure modernization

Replace unsupported Windows Server versions and aging SQL hosts. Move to supported, hardened, dedicated gateway infrastructure to reduce security, stability, and compliance risk.

3

Architecture standardization

Centralized, structured design — dedicated clusters with clear separation between production and non-production, defined ownership, and documented governance.

4

Connection rationalization

Replace many individual one-off connections with standardized, reusable connectivity patterns. Easier troubleshooting, better performance, less operational overhead.

5

Design + advisory engagement

Architecture workshops, implementation support, and a phased migration roadmap — including a cost / effort view for internal vs. Microsoft-assisted delivery.

6

Fabric adoption enablement

Gateway redesign is a prerequisite for scaling Microsoft Fabric adoption. Once the gateway tier is modernized, Direct Lake, Mirroring, and Dataflows Gen2 become viable patterns.

Additional considerations to include

Items beyond the initial six use cases that are easy to miss but materially affect delivery. Add these to the engagement scope up-front.

+

Migration roadmap

A phased approach that delivers quick wins early, de-risks the cutover, and clears the path for Fabric.

Phase Objective Key activities Outcome
1. Discover Build the full inventory and risk picture Catalog every gateway, host OS/version, connection, owner, refresh schedule, data source, downstream report. Identify duplicates and orphans. Inventory + heatmap
2. Design Lock the target architecture Cluster topology, VM sizing, network & firewall design, identity model, monitoring stack, recovery key escrow, governance model. Architecture pack
3. Build Stand up the new gateway estate Provision Windows Server 2022 hosts (PROD + Non-PROD clusters), install gateway, register to tenant, baseline monitoring, smoke-test connections. New estate live
4. Migrate Move connections in waves (parallel run) Migrate by workspace / business unit, re-point datasets, validate refresh parity, retire duplicates, document each owner sign-off. Connections rationalized
5. Decommission Retire legacy infrastructure Verify no traffic to old gateways for an agreed soak period, archive configuration, shut down legacy hosts, reclaim licenses. Legacy gone
6. Optimize Tune for Fabric & ongoing operations Enable Direct Lake / Mirroring where applicable, tune capacity, refine refresh windows, formalize governance, train admins. Fabric-ready

Quick wins (first 30 days)

Low-risk actions the customer can begin immediately, in parallel with the broader design work.

Inventory all gateways

Export from the Fabric admin portal: gateway names, members, host OS, version, connections, owners. This becomes the source of truth.

Identify the top 5 risks

Flag any gateway on an unsupported OS, any single-node cluster supporting prod reporting, and any connection without a named owner.

Stand up a PROD cluster on Server 2022

Two new dedicated nodes, registered as a cluster. Even before migration, this is the landing zone for everything that follows.

Enable gateway logging

Verbose gateway logs to Log Analytics. Establish the performance baseline now, so you can prove improvements after cutover.

Escrow the recovery key

Store the gateway recovery key in Azure Key Vault with documented access — most outages without it become full re-registrations.

Publish a connection request process

Stop the sprawl at the source. Every new connection now requires owner, environment, and data classification before it's provisioned.

Customer-facing talking points

Use these in exec and architect-level conversations.

Business

Modernizing the gateway tier isn't an IT project — it's how you unblock Fabric, reduce audit risk on unsupported OS, and stop paying the operational tax of one-off connections.

Technical

Dedicated HA gateway clusters with environment separation, central monitoring, and a documented connection lifecycle is the minimum bar for production analytics workloads in 2026.

Risk

Every additional month on Windows Server 2012 + mixed prod/non-prod hosts compounds security exposure, support risk, and the eventual cutover cost.

Path forward

A short discover + design engagement (architecture workshops) followed by a phased migration with parallel run gives the customer a low-risk path and a clear cost envelope.

Phase 1 · Plan
1

Prerequisites & access

half-day

Confirm everything you need is approved and provisioned before you touch infrastructure. Missing identity or firewall approvals are the #1 cause of cutover delays.

Actions

  1. Confirm Fabric / Power BI Service Administrator role for at least two named operators.
  2. Confirm Azure subscription Contributor (or scoped RG Owner) for VM provisioning and Key Vault.
  3. Confirm Active Directory rights to create a gMSA or service account in the agreed OU.
  4. Confirm the firewall team has the change record reference; outbound 443 rules to Azure Service Tags will need approval.
  5. Confirm sign-off on the target topology: 2 PROD nodes + 2 Non-PROD nodes (4 VMs total).
  6. Confirm an Azure Key Vault exists (or create one) in the same region for recovery key escrow.
Done whenAll access is granted in writing and verified by signing in to each portal.
2

Inventory the existing estate

1–2 days

You cannot migrate what you cannot list. Pull a definitive inventory from the Fabric admin portal first — this becomes the migration tracker.

Actions

  1. Run the Fabric admin PowerShell module against the tenant:
Install-Module -Name MicrosoftPowerBIMgmt -Scope CurrentUser
Connect-PowerBIServiceAccount

# All gateway clusters and members
Get-PowerBIGatewayCluster -Scope Organization |
  Select-Object Id, Name, Type, @{N='Members';E={ ($_.Members.gatewayName) -join ',' }} |
  Export-Csv .\gateways.csv -NoTypeInformation

# All gateway data sources
Get-PowerBIGatewayCluster -Scope Organization | ForEach-Object {
  Get-PowerBIDatasource -GatewayClusterId $_.Id -Scope Organization |
    Select-Object @{N='Cluster';E={$_.gatewayClusterId}}, Id, DatasourceName, DatasourceType, ConnectionDetails
} | Export-Csv .\datasources.csv -NoTypeInformation
  1. For each gateway host, capture: OS version, gateway version, host owner, environment (prod/non-prod), shared vs. dedicated.
  2. For each data source, capture: business owner, downstream datasets, refresh frequency, last successful refresh.
  3. Flag anything that hasn't refreshed in 90+ days as a decommission candidate.
Done whenA single spreadsheet exists with one row per connection, an owner name, and a migration wave assignment.
Phase 2 · Provision
3

Provision Windows Server 2022 hosts

half-day

Stand up four dedicated VMs. Two for the PROD cluster, two for Non-PROD. Match the baseline OS hardening you use for other Tier-1 hosts.

Recommended sizing (baseline)

  • SKU: Azure D8s_v5 (8 vCPU / 32 GB / Premium SSD) or equivalent on-prem.
  • OS: Windows Server 2022 Datacenter, fully patched.
  • Disk: 128 GB OS, 256 GB data disk for gateway cache (%LocalAppData%\Microsoft\On-premises data gateway).
  • Network: Static IP, joined to AD, NTP synced.

Example Azure CLI

RG=rg-fabric-gateway-prod
LOC=eastus2
for NAME in server1 server2; do
  az vm create \
    --resource-group $RG --location $LOC --name $NAME \
    --image MicrosoftWindowsServer:WindowsServer:2022-datacenter-azure-edition:latest \
    --size Standard_D8s_v5 \
    --vnet-name vnet-fabric --subnet snet-gateway \
    --admin-username gwadmin --admin-password '<set-in-keyvault>' \
    --public-ip-address "" \
    --nsg "" \
    --data-disk-sizes-gb 256
done
Why no public IPThe gateway only requires outbound 443. No inbound from the internet is needed or allowed.
Done whenAll four VMs are pingable from the gateway admin jumpbox, joined to the domain, and show as healthy in your monitoring tool.
4

Open the network path

depends on firewall team

The gateway communicates with the Power BI service using outbound HTTPS and Azure Relay. Don't open ports broadly — use Azure Service Tags.

Required outbound rules from each gateway host

  • TCP 443 → Service Tag PowerBI
  • TCP 443 → Service Tag AzureActiveDirectory
  • TCP 443 → Service Tag ServiceBus
  • TCP 443 → Service Tag AzureCloud (scoped to the home region if possible)
  • TCP 5671/5672 → AMQP fallback (only if you cannot use 443-only HTTPS mode)

Validation from the host

# PowerShell on each gateway host
Test-NetConnection -ComputerName api.powerbi.com -Port 443
Test-NetConnection -ComputerName login.microsoftonline.com -Port 443
Test-NetConnection -ComputerName <your-region>.servicebus.windows.net -Port 443
Done whenAll three TCP tests return TcpTestSucceeded : True. DNS resolves cleanly. No proxy auth prompts.
5

Create the gateway service identity

half-day

Run the gateway service under a dedicated identity. A group Managed Service Account (gMSA) is best — no password to rotate, no expiry surprises.

Create the gMSA

# On a domain controller (one-time, if not already done)
Add-KdsRootKey -EffectiveTime ((Get-Date).AddHours(-10))

# Create the gMSA and authorize the four gateway hosts
New-ADServiceAccount -Name gw-svc-prod `
  -DNSHostName gw-svc-prod.contoso.local `
  -PrincipalsAllowedToRetrieveManagedPassword "server1$","server2$","server3$","server4$"

# On each gateway host
Install-ADServiceAccount gw-svc-prod
Test-ADServiceAccount   gw-svc-prod   # must return True

SQL Server least-privilege logins

  • Create a SQL login per data source the gateway will read.
  • Grant db_datareader only (plus EXECUTE on specific stored procs if required). Never db_owner.
  • Document the mapping in the inventory spreadsheet.
Done whenTest-ADServiceAccount returns True on all four hosts and SQL logins authenticate from each host using sqlcmd -E.
Phase 3 · Install
6

Install the gateway on node A

30 min

Always install the standard mode gateway (not personal mode). This is the only mode that supports clustering and central administration.

On Server1 (PROD node A)

  1. Sign in interactively as a local admin (you'll switch the service to the gMSA after install).
  2. Download the latest installer from https://aka.ms/OnPremisesDataGateway (always pull the latest at install time).
  3. Run the installer, choose "On-premises data gateway (standard mode)", accept defaults for install path.
  4. Sign in with your Entra ID admin account when prompted.
  5. Choose "Register a new gateway on this computer".
  6. Name it clearly: cluster-prod-fabric.
  7. Set a strong recovery key (minimum 16 chars, generated by your password manager). Do not lose it.
  8. After install, switch the gateway service to the gMSA via Service account tab in the gateway app.
Done whenThe Fabric admin portal > Settings > Manage connections and gateways shows cluster-prod-fabric with status Online.
7

Escrow the recovery key to Key Vault

15 min · do not skip

The recovery key is the only thing that lets you restore the gateway if a node is lost. Without it, you have to re-register every connection from scratch.

az keyvault secret set \
  --vault-name kv-fabric-secrets \
  --name gateway-cluster-prod-recovery-key \
  --value '<the recovery key>' \
  --tags gateway=cluster-prod-fabric createdBy=<you> createdOn=2026-05-27

# Restrict access
az keyvault set-policy --name kv-fabric-secrets \
  --object-id <gateway-admins-group-objectid> \
  --secret-permissions get list

Document the access process

  • Who can retrieve the secret (named group, not individuals).
  • Under what scenarios (node loss, cluster rebuild).
  • Mandatory break-glass approval.
Done whenThe secret is in Key Vault, soft-delete is enabled, and a non-admin on the gateway-admins group can az keyvault secret show it.
8

Add node B and form the HA cluster

30 min

Two-node clusters give you load balancing and failover. A single-node "cluster" is not HA — never run prod on one node.

On Server2 (PROD node B)

  1. Run the same installer.
  2. Sign in with the same Entra ID admin account used on node A.
  3. Choose "Add to an existing gateway cluster".
  4. Select cluster-prod-fabric from the dropdown.
  5. Enter the recovery key retrieved from Key Vault.
  6. Switch the service to the same gMSA.
Done whenThe cluster shows 2 members, both Online, and load balancing is enabled in cluster settings.
9

Repeat for the Non-PROD cluster

1 hour

Repeat steps 6–8 on Server3 and Server4. Name the cluster cluster-nonprod-fabric. Use a different recovery key, escrow separately, use a Non-PROD gMSA.

Why separateA shared recovery key or shared service account couples your prod and non-prod blast radius. Keep them fully independent.
Done whenTwo clusters exist in the admin portal: cluster-prod-fabric (2 nodes) and cluster-nonprod-fabric (2 nodes), both Online.
Phase 4 · Operate
10

Wire up monitoring & logs

half-day

You cannot tune what you cannot see. Get gateway telemetry flowing to Log Analytics on day one — baselines collected now make every later perf claim defensible.

Enable additional logging

  1. In the gateway app on each node, go to Diagnostics.
  2. Enable Additional logging.
  3. Set Query execution & refresh history report to ON.

Forward to Log Analytics

  1. Install the Azure Monitor Agent (AMA) on each gateway host (via Azure Arc if on-prem).
  2. Create a Data Collection Rule (DCR) that picks up:
    • C:\Users\<gMSA>\AppData\Local\Microsoft\On-premises data gateway\Gateway*.log
    • Windows Performance Counters: CPU %, Memory available MB, Disk queue length
    • Windows Event Log: Application (filter Source = "Mashup*")

Alerts to create

  • Gateway cluster status = Offline for > 5 min → page on-call.
  • Refresh failure rate > 5% over 1 hour → email gateway-admins.
  • CPU > 85% sustained 15 min → email gateway-admins (capacity warning).
  • Heartbeat missed from a single node > 2 min → ticket (HA degraded, not down).
Done whenA Log Analytics workspace query like AzureDiagnostics | where Category == "GatewayLogs" returns rows from all four nodes within 15 minutes.
11

Create connections in the Fabric admin portal

1–2 days

Create one connection per data source on the new cluster — not one per dataset. This is where you eliminate the sprawl.

Per data source

  1. In Fabric > Manage connections and gateways > + New.
  2. Choose On-premises, select cluster-prod-fabric.
  3. Connection type: SQL Server / File / SharePoint / etc.
  4. Use the least-privilege SQL login created in step 5 (or Windows auth via the gMSA where supported).
  5. Name it deterministically: sql-{server}-{db}-prod, e.g. sql-finance-edw-prod.
  6. Set the connection to "Use SSO via Kerberos for DirectQuery" only if the source supports it and the business need is established.
  7. Grant connection users to the owning workspace's service principal / group (no individual user grants).
Done whenThe "Test connection" succeeds for every source, and a smoke-test report bound to the new connection refreshes successfully.
Phase 5 · Migrate & cut over
12

Migrate datasets in waves (parallel run)

multi-week

Never big-bang. Pick a wave (workspace or business unit), repoint its datasets to the new connection, and run old + new in parallel until you have parity sign-off.

For each dataset in the wave

  1. Open the dataset's settings in Fabric.
  2. Under Gateway and cloud connections, map each data source to the matching new connection (created in step 11).
  3. Trigger a manual refresh.
  4. Confirm refresh duration is within ±20% of the legacy baseline.
  5. Leave the old connection in place but unbound — that's the rollback path during the soak window.
  6. Email the dataset owner: "migrated, please validate within 5 business days".
Wave sizeKeep waves under ~20 datasets. Smaller waves catch problems before they compound and make rollback trivial.
Done whenEvery dataset in the wave has refreshed successfully against the new connection at least 3 times AND the owner has signed off in the tracker.
13

Validate parity & performance

per wave

You're proving two things: same numbers (data parity) and same or better speed (performance parity).

Data parity

  • For 3 representative reports per wave: compare row counts and a handful of aggregate measures (sum of sales, distinct customer count) between old-gateway snapshot and new.
  • Any drift > 0.1% must be root-caused before sign-off.

Performance parity

  • Capture refresh duration trends from Log Analytics over a 5-day window.
  • Capture DirectQuery query duration p95 from the Power BI usage metrics report.
  • Investigate any > 25% regression.
Done whenParity report attached to the migration ticket, signed by data owner and gateway lead.
14

Decommission legacy gateways

at end of soak

Only after a documented soak period (recommend 14 days of zero traffic on the old cluster) do you start shutting things down.

Steps

  1. Query gateway logs / usage metrics to confirm zero refreshes and zero DirectQuery hits on the legacy cluster for 14 consecutive days.
  2. In the admin portal, set the legacy cluster to Disable (don't delete yet).
  3. Wait 7 more days — gives any monthly / quarterly jobs a chance to surface.
  4. Export the legacy configuration (cluster definition, data source list) and archive it.
  5. Stop the gateway service, then uninstall.
  6. Shut down the legacy VMs. Wait 30 days, then delete VMs and reclaim storage/licenses.
Done whenLegacy cluster is removed from the admin portal, hosts are powered off, and the change record is closed with archived configuration attached.
Phase 6 · Optimize
15

Tune for Fabric & hand off to ops

ongoing

The build is done — but the platform isn't truly "delivered" until ops can run it without you.

Tuning

  • Refresh staggering: spread schedules across the hour to avoid 7:00 AM stampedes.
  • Incremental refresh: enable on any dataset > 1 GB or with > 30 day history.
  • Direct Lake / Mirroring: for sources that support it (Snowflake, SQL DB), consider removing the gateway hop entirely.
  • Capacity rightsizing: review Fabric capacity utilization weekly for the first month — scale up or autoscale based on actual.

Handover artifacts to operations

  • Architecture diagram (current target state).
  • Connection inventory + owner matrix (kept current).
  • Restart / failover / node-replacement runbook.
  • Recovery key escrow location and access process.
  • Monitoring alert list and on-call rotation.
  • Change-control process for new data source connections.
Done whenOps has run at least one rehearsed failover drill (drain node A, confirm node B carries the load, restore node A) without your involvement.
